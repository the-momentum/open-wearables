# Open Wearables - Mobile SDK Architecture Plan

## Context

Open Wearables supports **3 providers** (Apple Health, Google Health Connect, Samsung Health) across **2 cross-platform frameworks** (Flutter, React Native). The system consists of standalone native SDKs (iOS Swift SDK, Android Kotlin SDK) consumed by thin Flutter and React Native plugins. Each SDK collects health data from its respective platform, normalizes it into a unified payload format, and syncs it to the Python backend. This document describes the architecture and design decisions behind the current implementation.

---

## 1. Architectural Decisions

### 1.1 Type Mapping: Hybrid Approach (iOS Server-Side, Android Client-Side)

The type mapping strategy differs by platform:

- **iOS (Apple Health)**: The SDK sends HealthKit native identifiers as-is (e.g., `"HKQuantityTypeIdentifierHeartRate"`). The backend maps these to unified `SeriesType` values via `METRIC_TYPE_TO_SERIES_TYPE` in `backend/app/constants/series_types/apple.py`.
- **Android (Health Connect & Samsung Health)**: Both providers normalize types to unified SCREAMING_SNAKE_CASE names on the client side (e.g., `"HEART_RATE"`, `"STEP_COUNT"`, `"OXYGEN_SATURATION"`). Health Connect's `HeartRateRecord` class and Samsung's `DataTypes.HEART_RATE` both map to the same `"HEART_RATE"` string before upload. The backend distinguishes providers via the `provider` field in the payload, not the type string.

**Why this hybrid approach**:
- **iOS**: Server-side mapping is practical because HealthKit identifiers are stable, well-documented strings. Backend-side mapping lets us add new type support without an SDK release.
- **Android**: Client-side normalization is necessary because Health Connect and Samsung Health have fundamentally different APIs (class-based vs string-based) that are easier to unify in the native SDK layer. Both providers share the same unified type names, simplifying the backend mapping to a single Android mapping module.
- **Unit conversion stays client-side for all platforms**: The iOS SDK converts via `_defaultUnit()` in `Types.swift`. The Android SDK normalizes units during record conversion in `HealthConnectManager.kt` and `SamsungHealthManager.kt`.

**Trade-off**: The backend needs two mapping paths - one for Apple's native identifiers, one for Android's unified identifiers. The `provider` field in the payload determines which mapping to use.

### 1.2 Payload Format: Unified Schema for All Providers

**Recommendation**: All providers send data in the same unified schema. The backend routes by provider (from URL path) and applies provider-specific type mappings.

**Current state**: The sync endpoint already routes by provider (`/sdk/users/{user_id}/sync/{provider}`). The existing Apple schema serves as the foundation for the unified format.

**Key design principles**:
- **One schema, all providers**: Same JSON structure regardless of whether data comes from Apple Health, Google Health Connect, or Samsung Health. Field names and nesting are identical.
- **Types are strings, but differ by platform**: iOS sends HealthKit native identifiers (`"HKQuantityTypeIdentifierHeartRate"`). Android sends unified SCREAMING_SNAKE_CASE names (`"HEART_RATE"`) for both Health Connect and Samsung Health. Backend uses the `provider` field to select the correct mapping.
- **Sleep supports optional `parentId`**: Providers with native sleep sessions (Google HC, Samsung) send a parent session record + child stage records linked via `parentId`. Providers without sessions (Apple) send flat phase records without `parentId` -- backend reconstructs sessions via existing Redis state machine.

**Backend structure**:
```
backend/app/
  schemas/
    sdk/
      sync_request.py               # UNIFIED schema for all providers
  constants/series_types/
    apple.py                         # existing - HK identifier → SeriesType
    google.py                        # new - HC record type → SeriesType
    samsung.py                       # new (if separate from Google)
  constants/workout_types/
    apple_sdk.py                     # existing
    google.py                        # new
  services/
    sdk/
      import_service.py              # unified, receives provider as param
      sleep_service.py               # updated: handles parentId + Redis paths
  integrations/celery/tasks/
    process_apple_upload_task.py      # existing
    process_google_upload_task.py     # new
    process_samsung_upload_task.py    # existing, extend
```

**Why unified over provider-specific schemas**: A single schema means one validation path, one set of field names, and one documentation source. Provider-specific interpretation happens only in the mapping layer (constants), not in schema or serialization. Debugging remains easy -- the `type` strings tell you exactly what the device sent.

### 1.3 Proposed Unified Payload Format

#### Top-level structure

```json
{
  "provider": "apple_health | samsung_health | health_connect",
  "sdkVersion": "0.10.0",
  "syncTimestamp": "2024-01-15T12:00:00Z",
  "data": {
    "records": [ ... ],
    "sleep": [ ... ],
    "workouts": [ ... ]
  }
}
```

All three arrays are optional -- a sync batch can contain any combination. The `provider` field determines which type mapping the backend applies.

#### records (time-series measurements)

```json
{
  "id": "0F12CD84-80C1-45D2-A0CD-474C144602C4",
  "type": "HKQuantityTypeIdentifierHeartRate",
  "startDate": "2024-01-01T03:54:07Z",
  "endDate": "2024-01-01T03:57:20Z",

    "appId": "com.apple.health",
    "deviceId": null,

    "deviceManufacturer": "Apple Inc.",
    "deviceModel": "Watch7,3",

  "value": 72,
  "unit": "bpm",
  "parentId": null,
  "metadata": { "HKMetadataKeyHeartRateMotionContext": "1" }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Stable unique ID from provider (HealthKit UUID, HC `Record.metadata.id`) |
| `type` | string | yes | Type identifier (Apple: HK native, Android: unified SCREAMING_SNAKE_CASE) |
| `startDate` | ISO 8601 | yes | Sample start time |
| `endDate` | ISO 8601 | yes | Sample end time |
| `zoneOffset` | string | no | Timezone offset (e.g., `"+01:00"`). Null for Apple (always UTC). |
| `source` | object | no | Device/app info (see source section) |
| `value` | number | yes | Measurement value in standardized units |
| `unit` | string | yes | Standardized unit (bpm, m, kg, kcal, etc.) |
| `parentId` | string | no | Links split records back to parent (e.g., HR series samples, blood pressure components) |
| `metadata` | object | no | Provider-specific key-value pairs |

**`type` examples per provider**:
- **Apple**: `"HKQuantityTypeIdentifierHeartRate"`, `"HKQuantityTypeIdentifierStepCount"`, `"HKQuantityTypeIdentifierBodyMass"`
- **Google HC**: `"HEART_RATE"`, `"STEP_COUNT"`, `"OXYGEN_SATURATION"` (unified names, same as Samsung)
- **Samsung**: `"HEART_RATE"`, `"STEP_COUNT"`, `"BLOOD_OXYGEN"` (unified names, same as Google HC)

#### sleep (stages with optional session grouping)

Each entry represents **one sleep stage** (not a session). The SDK flattens provider-specific nested structures into flat stage records. Sessions are grouped via `parentId`.

**Why payloads differ per provider -- and why `parentId` is optional:**

- **Apple HealthKit** has no concept of a "sleep session". It stores sleep as individual `HKCategorySample` records with `value` (0=IN_BED, 1=ASLEEP_UNSPECIFIED, 2=AWAKE, 3=LIGHT, 4=DEEP, 5=REM). The iOS SDK maps these int values to unified stage strings (`"in_bed"`, `"sleeping"`, `"awake"`, `"light"`, `"deep"`, `"rem"`). No `parentId` -- backend reconstructs sessions via Redis-based gap analysis.

- **Google Health Connect** natively models sleep as a `SleepSessionRecord` with `Stage` objects. The Android SDK flattens these into individual stage records with `parentId` linking back to the session ID.

- **Samsung Health SDK** follows a similar nested pattern. The SDK flattens `sessions[].stages[]` into individual records with `parentId`.

### 1.4 Core Logic: Heavy Native, Thin Bridge

All health data collection, background sync, serialization, upload, and outbox logic lives in standalone native SDKs. Flutter/RN are thin bridges (~200 lines each).

**Architecture**:
```
[iOS Swift SDK]                        [Android Kotlin SDK]
open_wearables_ios_sdk                 open_wearables_android_sdk
(HealthKit, BGTaskScheduler,           (Health Connect, Samsung Health,
 anchor sync, outbox, keychain)         WorkManager, Room outbox, Keystore)
       |                    |                  |                    |
[Flutter iOS Plugin]  [RN iOS Module]  [Flutter Android Plugin]  [RN Android Module]
(~200 lines Swift)    (~200 lines Swift) (~200 lines Kotlin)    (~200 lines Kotlin)
       |                    |                  |                    |
[Flutter Dart API]    [RN TypeScript API]  [Flutter Dart API]   [RN TypeScript API]
```

**iOS**: The Swift SDK (`open_wearables_ios_sdk/`) is a standalone CocoaPod (~> 0.10.0). The Flutter plugin (`OpenWearablesHealthSdkPlugin.swift`, ~200 lines) delegates to `OpenWearablesHealthSDK.shared`.

**Android**: The Kotlin SDK (`open_wearables_android_sdk/`) is a standalone library with two provider implementations - `HealthConnectManager` for Google Health Connect and `SamsungHealthManager` for Samsung Health. On Samsung devices, the SDK auto-selects Samsung Health by default, but users can override via `setProvider()`. The Flutter plugin delegates to the native SDK.

## 3. Android Native SDK

The Android Kotlin SDK (`open_wearables_android_sdk`) supports two providers via a `HealthDataProvider` interface:

```
open_wearables_android_sdk/
  sdk/src/main/kotlin/
    OpenWearablesHealthSDK.kt          # Main singleton
    HealthConnectManager.kt            # Google Health Connect provider
    SamsungHealthManager.kt            # Samsung Health provider
    SyncManager.kt                     # Orchestration, chunking, auth retry
    UnifiedPayload.kt                  # Data models for unified payload
    SecureStorage.kt                   # EncryptedSharedPreferences
    HealthSyncWorker.kt                # WorkManager background sync
```

**Key features**:
- **Dual provider**: `HealthDataProvider` interface abstracts over `HealthConnectManager` and `SamsungHealthManager`. Auto-selects Samsung Health on Samsung devices, with `setProvider()` override.
- **Round-robin sync**: `SyncManager` fetches one chunk from each data type, merges into a combined payload, sends in a single HTTP request. Chunk size is 2000 items total.
- **Streaming JSON**: Uses `JsonWriter` to stream JSON directly to the network socket - O(depth) memory, not O(n).
- **Background sync**: `WorkManager` with foreground service (type `health`), min 15-min interval.
- **Incremental sync**: Health Connect `getChangesToken()` / `getChanges()`, Samsung Health timestamp-based anchoring.
- **Auth**: Dual mode - Bearer token (with automatic refresh via `/api/v1/token/refresh`) or API key (`X-Open-Wearables-API-Key` header). Credentials stored in `EncryptedSharedPreferences`.
- **Resumable sync**: Sync state persisted to disk. If interrupted, resumes from where it left off.

**Key differences from iOS**:
- **Background sync**: `WorkManager` with `PeriodicWorkRequest` instead of `HKObserverQuery` + `BGTaskScheduler`
- **Incremental sync**: Health Connect `getChangesToken()` / `getChanges()` instead of `HKAnchoredObjectQuery`
- **Outbox**: Room database instead of file-based
- **Secure storage**: Android Keystore + `EncryptedSharedPreferences` instead of iOS Keychain
- **Permissions**: Health Connect returns explicit `GRANTED`/`DENIED` (unlike HealthKit's privacy-by-design where you can't know)

## 4. Additional Considerations

### 4.1 Payload Compression
Large initial exports can be megabytes. Add gzip:
- iOS: `Data.compressed(using: .zlib)`, set `Content-Encoding: gzip`
- Android: `GZIPOutputStream`
- FastAPI handles decompression automatically

### 4.2 SDK Version Header
Send `X-SDK-Version: flutter-ios/1.2.0` (or `rn-android/1.0.0`) with every API request. Backend logs and stores it per batch. Enables:
- Diagnosing version-specific bugs
- Deprecating old SDK versions (return `426 Upgrade Required`)

### 4.3 Remote Configuration Endpoint
```
GET /api/v1/sdk/config
{
  "min_sdk_version": "1.1.0",
  "sync_chunk_size": 2000,
  "telemetry_enabled": true,
  "supported_providers": ["apple", "google_health_connect"]
}
```
Allows tuning SDK behavior without releases (chunk sizes, feature flags, deprecations).

### 4.4 Sleep Data Normalization

Sleep formats differ across providers:
- **Apple**: Individual `HKCategoryTypeIdentifierSleepAnalysis` records with phase values 0-5. No native session concept.
- **Google Health Connect**: `SleepSessionRecord` with `SleepSessionRecord.Stage` objects. Native parent-child structure.
- **Samsung**: Session with detailed stages (similar to Google HC).

**Unified approach (flat + parentId)**:

All providers use the same sleep array in the unified payload. Providers with native sessions (Google HC, Samsung) send a parent session record + child stage records linked via `parentId`. Apple sends flat phase records without `parentId`.

**Backend handles both paths**:
1. **Structured path** (records have `parentId`): Build parent→children map. Create `EventRecord + SleepDetails` directly by aggregating child stage durations. No Redis needed.
2. **Flat path** (no `parentId`, orphan records): Existing Redis-based sleep state machine reconstructs sessions from individual phases. Detects session boundaries via gap analysis (>1 hour gap = new session).

Both paths produce identical `EventRecord + SleepDetails` output. See Section 1.3 for payload examples.

### 4.5 Health Connect Permissions Model
Apple HealthKit returns `true` for `requestAuthorization()` even if the user denied (privacy by design). Health Connect returns explicit `GRANTED`/`DENIED`. The platform interface should account for this:
```dart
enum AuthorizationResult {
  granted,
  denied,
  undetermined,  // iOS -- cannot determine if granted
}
```

### 4.6 Deduplication
Already handled via `external_id` (UUID from HealthKit) with `ON CONFLICT DO NOTHING` on the backend. Ensure every provider sends stable unique IDs per record (Health Connect provides `Record.metadata.id`).

### 4.7 Rate Limiting & Exponential Backoff
SDK should implement backoff for failed uploads: 5s -> 30s -> 5min -> rely on periodic background sync. Backend should return `429` with `Retry-After` header if syncing too aggressively.
